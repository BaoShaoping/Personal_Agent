# 托管 Beta 设计：无状态代理 + 浏览器端数据

> 蓝图存档，**尚未实现**。把「System Edition v0（本地单用户，`v0.2.0`）」推向「普通用户一个链接就能测」的托管 Beta，用于观察真实用户反馈（closed beta → 再放大，像游戏内测）。
> 设计哲学：**服务端什么都不存**；用户成长数据留在**浏览器**里；服务端只是一个**瘦的、无状态的 GLM 代理**。即使托管，**local-first 依然保住**。

## 0. 一句话
浏览器 = 数据的唯一真相（localStorage）。服务端 = 一个无状态瘦代理：托管静态页 + 代理 GLM 调用（持 key、限 3 并发、用 glm-4.5-air）。邮箱只用外部收集器登记测试者名单，服务端零存储。

## 1. 已锁定的决定
- **身份/邮箱**：方案 (a) —— 邮箱仅用于**测试者名单 + 联系**（外部收集器），**App 数据全在浏览器**，与邮箱不绑定。
- **公开版模型**：`glm-4.5-air`（快/省）。按部署用环境变量覆盖；本地开发仍 `glm-4.6`。
- **不存储 / 暂无服务器**：服务端零持久化；只需一个轻量无状态代理（可小主机 / 以后甚至 serverless）。
- **key 并发限制：最多 3 个并发请求** → 代理内置 **≤3 并发队列**。

## 2. 架构
```
浏览器 (Client)                         瘦代理 (Server, 无状态)
─────────────                          ──────────────────────
localStorage = 真相                     GET  /            静态 SPA
  level/exp/五维/magic/                  POST /api/glm     GLM 代理：
  forest/avatar/unlocked/                  - 持 GLM key（客户端永不接触）
  plans/tasks/dings                        - ≤3 并发队列 + 限流 + 成本封顶
面板全部客户端渲染 ←──────────────────────  - 模型由 env 决定（公开=4.5-air）
（logic 已存在于 system.js）              不存任何用户数据
```
- **客户端**：所有成长数据在 localStorage；用 `system.js` **已有的**逻辑（`levelInfo`/`attrLevelInfo`/`forestStage`/`settleLocally`/各 render）直接计算与渲染。**这是关键省力点**——这些代码在 step 1 / fallback 时期就写好了。
- **服务端**：只做两件事——托管静态页、代理 GLM。**无状态、零存储、无每用户数据**。
- **邮箱**：外部收集器（Formspree / Google Form / 类似），服务端不碰。

## 3. 数据流（相对本地版是反转的）
- **打开面板**：从 localStorage 读 state（首次运行用默认 seed）→ 客户端渲染。**不需要调服务端**。
- **完成任务**：客户端本地结算（`settleLocally`：经验/魔法点/属性/森林）→ 写 localStorage → 「叮！」（旁白用模板，或可选调 `/api/glm` 让 LLM 写）。
- **生成任务**：客户端把计划上下文（来自 localStorage）发给 `/api/glm` → GLM(4.5-air) → 任务 → 客户端显示建议 → 接受 → 写入 localStorage 的今日任务。
- **结论**：服务端**只在 GLM（任务生成 / 可选旁白）时被触达**，其余全客户端。

## 4. 并发与成本护栏（围绕「≤3 并发」这条硬约束）
- **代理并发闸**：一个异步信号量，最多 3 个在途 GLM 请求；第 4+ 个**排队等待**（客户端显示「排队中…」），不报错。
- **限流**：按 session/IP 限速（如每分钟 N 次），防单用户刷爆你的 key。
- **成本封顶**（可选）：超预算后停止新 GLM 调用，**优雅降级**到规则任务 + 模板旁白（这些 fallback 我们已经有）。所以队列满/预算尽时不会硬失败。
- **模型** = `glm-4.5-air`（公开代理），快、便宜。

## 5. 邮箱（方案 a）
- **首屏门槛**：「输入邮箱开始测试」→ POST 到外部收集器（如 Formspree）→ 成功后在 localStorage 置一个标记 → 进入 App。
- 邮箱用途：**测试者名单 + 反馈联系**，**不与成长数据绑定**（数据在浏览器里是匿名的）。
- 服务端零存储。

## 6. 隐私（这版其实很强，要讲出来）
- 成长数据**永不离开浏览器**（localStorage）；服务端不存任何东西。
- 只有**任务生成/旁白的 prompt**（含部分计划上下文）会经代理发往 GLM——这点要明确告知。
- 邮箱只去外部收集器。

## 7. 反馈机制
- 站内「反馈」按钮 → 外部表单（同一或另一个收集器）。
- 可选：客户端本地的匿名事件计数，用户可一键提交（仍不强制服务端存储）。

## 8. 代码改造（build plan）
**好消息：客户端逻辑大半已存在**（stub/fallback 时期：`levelInfo`/`attrLevelInfo`/`forestStage`/`settleLocally`/renderers + STUB 的 state 形状）。转变主要是：
1. **localStorage 作真相**：浏览器加载/保存一个 state 对象；首次运行写默认 seed。把 `/api/system/summary` 的 fetch 换成读 localStorage；把各 mutation 的 POST 换成本地更新 + 客户端结算（`settleLocally` 升级为主路径，不再是 fallback）。
2. **服务端瘦身**：只保留静态托管 + `/api/glm` 代理（key + ≤3 并发 + 限流 + env 模型）。本地版的 `/api/system/*` 端点在托管版不再使用。
3. **两套构建模式**：**本地单用户**（现有 Flask + 文件，glm-4.6，开发用）与**托管版**（客户端数据 + 瘦代理，glm-4.5-air）。建议都保留——本地版方便开发/演示。
4. **任务/旁白经代理**：客户端用 localStorage 的上下文拼 prompt，调 `/api/glm`；解析放客户端或代理。
5. **邮箱门槛 + 反馈**：外部收集器。
6. **并发队列**：代理里实现。

### 分阶段
- **Phase 1（纯客户端，无需服务器）**：数据搬进 localStorage，客户端结算/渲染为主路径；App 在 mock/规则下**完全本地可用**（这一阶段甚至不需要任何服务器，可先验证体验）。
- **Phase 2（瘦代理）**：实现 `/api/glm`（key + ≤3 并发队列 + 限流 + env 模型 4.5-air），任务生成/旁白接上真 GLM。
- **Phase 3（发布）**：邮箱门槛 + 反馈（外部）+ 隐私说明 + 部署到一个小主机 → 拿到链接 → closed beta。

## 9. 待定
- **托管平台**（暂无服务器）：代理很轻，等就绪时挑个小主机 / serverless。
- **外部收集器**：Formspree / Google Form / 其它。
- **成本封顶数值** 与 **限流阈值**。
- 是否保留本地单用户 Flask 版并行（建议是，开发/演示用）。

## 10. 与现有架构的关系
- **复用** `system.js` 里已有的客户端计算与渲染（来自 stub/fallback 时期）——这是本方案比「服务端多用户数据库」轻得多的根本原因。
- Python 的 `system_engine` 等**保留给本地单用户版**；托管版主要靠客户端 JS + 瘦代理。
- 基线：本地单用户完整版 = tag `v0.2.0`。托管 Beta 是下一章（朝 `v0.3.0`）。
